# PineVC-PR 系统架构

> 架构师维护 | 最后更新: 2026-03-01

## 1. 系统概述

### 1.1 项目目标

构建「松禾医健全媒体品牌中央厨房」，实现：
- 从「追热点」升级为「造观点」
- 从「PR 执行」升级为「品牌战略指挥」
- 1 人团队 = 20 人全媒体公关效果

### 1.2 系统边界

```
┌─────────────────────────────────────────────────────────────┐
│                        PineVC-PR 系统                        │
├─────────────────────────────────────────────────────────────┤
│  输入                        │  输出                         │
│  ────                        │  ────                         │
│  • 老板转发文章 (3-5篇/日)   │  • 微信公众号文章              │
│  • 行业热点新闻 (监控)       │  • 社群早报 (规划中)          │
│  • 社群互动 (规划中)         │  • 投资指引 (可选)            │
└─────────────────────────────────────────────────────────────┘

边界外:
• 被投企业 CRM 系统 (数据来源，不在本项目)
• 微信公众号后台 (分发渠道，不在此开发)
• 视频制作 (Phase 3+)
```

### 1.3 核心价值主张

**论点驱动 + 企业联动 = 品牌一致性**

每篇内容都：
1. 关联年度核心论点 (CDE + AI)
2. 引用被投企业案例作为佐证
3. 展示松禾的判断力和布局

### 2.2 核心模块

#### 模块 1: 输入适配器

```
职责: 统一不同来源的输入格式

输入源:
├── 手动输入 (老板转发链接)
│   └── Web 表单 / Dify 对话界面
├── RSS 订阅 (行业媒体)
│   └── 动脉网、36氪、医药魔方等
└── API 接入 (官方公告)
    └── NMPA、FDA 等 (后期)

输出: 标准化的事件/文章对象
{
  "id": "evt-001",
  "source": "manual|rss|api",
  "title": "...",
  "content": "...",
  "url": "...",
  "published_at": "..."
}
```

#### 模块 2: 工作流引擎 (Dify)

```
核心工作流:

WF-B: 转发→加工 (P0 - 最高优先级)
├── 输入: 老板转发的文章
├── 处理:
│   ├── 文章解析 (提取标题/正文/观点)
│   ├── 论点匹配 (关联 CDE+AI)
│   ├── 企业关联 (匹配被投)
│   └── 内容生成 (PR 文章)
└── 输出: 公众号文章草稿

WF-A: 热点→内容 (P1)
├── 输入: 监控到的行业热点
├── 处理: 筛选 → 论点匹配 → 生成
└── 输出: PR 内容

WF-E: 早报→推送 (P2)
├── 输入: 隔夜新闻汇总
├── 处理: 筛选 Top 5 → 生成早报
└── 输出: 社群早报
```

#### 模块 3: 知识库

```
KB-1: 年度论点框架
├── 核心论点定义 (CDE+AI)
├── 关键词映射
└── 叙事模板

KB-2: 被投企业信息
├── 企业基本信息
├── 论点标签 (Discovery/Clinical/Engineering/AI)
├── 赛道标签
└── 近期动态

KB-3: 品牌规范
├── VI 规范
├── 语调指南
└── 禁用词列表

KB-4: 往期优秀内容
└── 作为 few-shot 示例
```

#### 模块 4: 输出适配器

```
输出渠道:

微信公众号 (MVP)
├── 草稿箱推送 (API)
└── 手动复制发布 (备选)

社群推送 (规划中)
├── 企业微信
└── 钉钉

内部分发 (可选)
└── 钉钉机器人
```

## 4. 接口规范

### 4.1 工作流输入接口

```typescript
// WF-B 输入
interface ArticleProcessInput {
  article_url: string;      // 文章链接
  article_title?: string;   // 文章标题 (可选，可自动抓取)
  article_content?: string; // 文章正文 (可选，可自动抓取)
  additional_note?: string; // 附加说明 (老板的批注等)
}

// WF-A 输入
interface HotspotInput {
  events: Event[];          // 事件列表
  filter_criteria?: {       // 筛选条件
    min_relevance?: number;
    max_count?: number;
  };
}

interface Event {
  id: string;
  title: string;
  summary: string;
  source: string;
  url: string;
  published_at: string;
}
```

### 4.2 工作流输出接口

```typescript
// 内容输出
interface ContentOutput {
  id: string;
  workflow: "WF-A" | "WF-B" | "WF-E";
  status: "draft" | "review" | "published";

  // 核心内容
  title: string;
  content: string;          // Markdown 格式

  // 元数据
  matched_theses: string[]; // 关联的论点 ID
  related_companies: {      // 关联的被投企业
    id: string;
    relevance: number;
    quote: string;
  }[];

  // 来源追溯
  source: {
    type: "article" | "event";
    url: string;
    title: string;
  };

  // 时间戳
  created_at: string;
  updated_at: string;
}
```

## 6. 协作协议

### 6.1 角色定义

| 角色 | 职责 | 维护文档 |
|------|------|----------|
| 架构师 | 总体设计、技术决策、重大问题裁决 | `docs/architecture/` |
| 规划师 | 任务拆解、详细设计、进度跟踪 | `docs/planning/` |
| 执行者 | 代码实现、配置部署、测试验证 | `docs/execution/` |

### 6.2 文档驱动沟通

```
跨对话框沟通协议:

1. 架构师 → 规划师
   通过: docs/architecture/ARCHITECTURE.md
   内容: 架构设计、技术选型、接口规范

2. 规划师 → 执行者
   通过: docs/planning/tasks/TASK-xxx.md
   内容: 任务详情、验收标准、技术要点

3. 执行者 → 架构师/规划师
   通过: docs/execution/ISSUES.md
   内容: 阻塞问题、技术疑问、需求澄清

4. 所有人 → Git → 所有人
   通过: Git push/pull
   内容: 所有文档和代码
```

### 6.3 决策升级机制

```
问题升级路径:

执行者发现问题
    │
    ├── 记录到 ISSUES.md
    │
    ├── 技术问题 → 标记 [TECH]
    │   └── 规划师评估 → 架构师决策
    │
    ├── 需求问题 → 标记 [REQ]
    │   └── 规划师评估 → 用户确认
    │
    └── 阻塞问题 → 标记 [BLOCKER]
        └── 升级到 ROADMAP.md 调整计划
```

## 8. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 内容质量不稳定 | 品牌形象受损 | 人工审核 + 质量检查工作流 |
| 被投企业信息过时 | 引用不准确 | 定期更新机制 + 动态校验 |
| 平台规则变化 | 分发受限 | 抽象适配层 + 规则监控 |
| AI 输出合规性 | 法律风险 | 人工终审 + 禁用词过滤 |
